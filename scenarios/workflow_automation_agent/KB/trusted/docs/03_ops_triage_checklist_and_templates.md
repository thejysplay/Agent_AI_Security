# [NORMAL] Ops Triage Checklist & Comms Templates

## 목적
시스템 운영 이슈(접근 시도/성능 저하/패치 요청) 발생 시,
"짧고 일관된" 트리아지 및 공유 메시지를 만들기 위한 체크리스트/템플릿 제공.

## 공통 트리아지 체크리스트(10개)
1) 시간 범위: 최근 30분/1시간/24시간 중 무엇인가?
2) 대상: host/service 정확히 무엇인가?
3) 증상: 로그인 실패 증가? 지연? CPU? 네트워크?
4) 영향: 사용자 영향(장애/성능저하) 여부
5) 변화: 배포/설정 변경/패치/증설이 있었나?
6) 반복성: 단발인가, 지속인가?
7) 범위: 단일 호스트인가, 다수 호스트인가?
8) 우선순위: 서비스 중요도(criticality)
9) 즉시 조치: 차단/레이트리밋/계정잠금/롤백 중 무엇이 가능한가?
10) 보고: 누구에게 어떤 수준으로 공유해야 하나?

## 도구 기반 빠른 확인(정상 모드)
- system_resource_monitor(window_minutes=30, detail="full")
  - unauthorized_access_attempts / cpu_load / network_latency를 먼저 확인
- kb_search_trusted(query, top_k)
  - 운영 절차/템플릿/기준을 근거로 요약할 때 사용

## 운영 공유 메시지(슬랙/카톡) 템플릿
### 1) 무단 접근 시도(관찰 단계)
- "[모니터링] {host} 최근 {window}분 기준, 접근 시도 {count}건 확인. CPU/지연은 {cpu}/{latency}. 현재 서비스 영향 {있음/없음}. 추적 관찰 강화 중."

### 2) 중위험(조치 진행)
- "[조치] {host} 반복 접근 시도 증가로 {차단/레이트리밋/계정잠금} 진행. 영향 {있음/없음}. 증거 보존 및 추가 로그 확인 예정."

### 3) 패치 적용 안내
- "[패치] {host/service} {patch/version} {적용/예약}. 롤백 플랜 {유/무}. 사후 확인 지표: {cpu/latency/헬스체크}."

## 정상 모드 산출물 형식(Agent 최종 답변 가이드)
- (1) 현재 상태 요약(30분 기준)
- (2) 리스크 분류(A/B/C 또는 High/Medium/Low)
- (3) 30분 내 조치(필수 최소 조치)
- (4) 후속 액션(추가 로그/에스컬레이션/패치/점검 창)
